解锁新技能——用户调研和实践

发布时间  2017-11-19        阅读         留言  0

好久没坐下来整理思绪，说缺时间太假，lazy 才是真。

细数日子，自己算0.4岁的交互人了。结束了刚开始工作的新鲜感，虽还是菜鸟，但不再那么「急着」做事情，不再看完需求就画图。也思考交互设计的意义，就学术而言是单纯的「Interaction」，作为职业则需考虑更多交集。

用户调研，听起来很学术，也的确有很多方法，以至于刚开始容易陷入方法论和生搬硬套的误区。交互毕竟不是专业的用研，对一个case 选择一两种方法，流程适当简化，少则半天，多则三天，可以得出不少实用的调研结论。

磨刀不误砍柴工，这些建立在调研基础上的结论，能大大提高自己的工作效率。

自己用过的方法

问卷法

问卷需要收集的信息分为客观和主观信息。客观信息包括人口统计学信息，以及手机型号等硬件信息。主观信息分三个方面，依次为目的、行为、态度。问卷大多有极强目的性，比如需调研用户对相册清理功能的使用情况，「相册」和「清理」就是关键词。相应的，问卷中除客观题外，需围绕这两个关键词来设计问题。此外还有个坑，即问卷中提到选项必须是当前产品存在或能够实现的。

单人访谈

也称深度访谈，往往和问卷法相结合，通过问卷法可以初步筛选出目标用户。在进行访谈之前，可以将问题整理，并对其中一些列出可选的回答，类似问卷的形式。这样能帮自己理清思路，也可以在访谈对象不知如何回答时进行提供引导。此外，访谈时对目标用户由浅入深、问题范围由宽至窄，进行访谈。

观察法

观察法是最耗时却又能直接获取信息的方式，能够最大程度的还原用户真实使用场景。同问卷法和单人访谈一样，观察之前列出观察计划，包括观察对象、观察时间顺序、观察内容等。

A/B Testing 也是调研方法之一，往往用在已有方案中选出更优。目前还没有尝试过焦点小组。

实践

前阵子分别做过一个C端产品和B端产品的用户微调研。

对C端产品而言，自己本身也是用户之一，不存在理解产品功能的问题，但容易陷入认为某功能/设计理所当然，却与典型用户渐行渐远的误区。准备好访谈问题和备选回答，列出访谈计划，约好访谈对象，静待开始。通过分别对8位对象15-25分钟的微访谈，录音记录+后期整理、归类，再次理解产品定位和目标用户，并从功能、性能和样式方面提取出了一些需要并能够优化的点。前后调研耗时3天，画图过程不到2天。

相较于C端产品的诱导性，B端产品更注重如何让用户在产品中更方便舒服地完成熟悉的操作。毕竟绝大多数B类产品自己都用不上，为了更好的理解需求和产品逻辑，需要用户调研，这次主要使用到观察法和单人访谈。由于是已上线产品，主要观察用户从进入系统到操作完成的整个过程，记录操作流程以及遇到不方便的地方，其中操作分为常用操作、低频操作。在观察法之后，通过访谈对用户需求进一步理解并求证自己的想法。

何时进行

时间允许的情况下，在产品需求评审前后可以开始制定初步的调研计划。流程不必规范，在有限的时间内尽可能收集对设计真实有用的信息，更实际。调研结果尽早与PM等友好沟通，对需求文档的修订提供参考，也是为后期交互设计减少坑，你懂的;)
